# Care Keeper

Care Keeper เป็นระบบส่วนติดต่อผู้ใช้แบบกราฟิก (Graphical User Interface: GUI) สำหรับเครื่องตรวจวัดสัญญาณชีพที่ทำงานบน Raspberry Pi พัฒนาด้วยภาษา Python และ PySide6 โดยออกแบบหน้าจอสำหรับอุปกรณ์แสดงผลขนาด 1010 x 503 พิกเซล ระบบนี้มีหน้าที่ช่วยให้ผู้ใช้งานสามารถอ่านข้อมูลผู้รับบริการจากบัตรประชาชน วัดค่าสัญญาณชีพ แสดงผล สรุปผล และส่งข้อมูลไปยังระบบ Backend ได้ภายในโปรแกรมเดียว

เอกสารนี้จัดทำเพื่อใช้เป็นทั้งคู่มือประกอบการติดตั้ง การทดสอบ และเป็นข้อมูลสำหรับอธิบายในรายงาน/การนำเสนอ

## วัตถุประสงค์ของโครงการ

เป้าหมายของโครงการคือการพัฒนา GUI สำหรับเครื่องตรวจสุขภาพเบื้องต้น เพื่อรวมการทำงานของอุปกรณ์หลายชนิดให้ผู้ใช้งานสามารถใช้งานได้ง่ายขึ้น ลดความซับซ้อนของการอ่านค่าจากอุปกรณ์จริง และแสดงผลลัพธ์ในรูปแบบที่เข้าใจง่ายสำหรับผู้ใช้ทั่วไป

ระบบรองรับการทำงานหลักดังนี้

- อ่านข้อมูลผู้รับบริการจากบัตรประชาชนไทย
- รองรับการกรอกเลขบัตรประชาชนเองในกรณีเครื่องอ่านบัตรมีปัญหา
- วัดค่าความดันโลหิต (SYS/DIA)
- วัดค่าชีพจร (Pulse)
- วัดค่าออกซิเจนในเลือด (SpO2)
- แสดงค่าอุณหภูมิร่างกาย
- แสดงสถานะ Wi-Fi, Bluetooth และแบตเตอรี่
- แสดงหน้าสรุปผลการวัด
- แสดงข้อมูลย้อนหลัง 4 รายการล่าสุด
- ส่งข้อมูลผลการวัดไปยัง Backend API

## ภาพรวมหน้าจอของระบบ

ระบบแบ่งหน้าจอหลักออกเป็น 3 หน้า

### 1. หน้าอ่านบัตรประชาชน

เป็นหน้าแรกของระบบ ใช้สำหรับเริ่มต้นการตรวจ โดยผู้ใช้สามารถกดปุ่มอ่านข้อมูลจากบัตรประชาชน หากเครื่องอ่านบัตรทำงานได้ปกติ ระบบจะอ่านเลขบัตร ชื่อ วันเกิด และที่อยู่ แล้วเข้าสู่หน้าวัดค่าสัญญาณชีพ

ในกรณีที่เครื่องอ่านบัตรมีปัญหา ระบบมีปุ่ม `กรณีอ่านไม่สำเร็จ กรุณากรอกเลขบัตรเอง คลิกที่นี่` เมื่อกดแล้วจะเปิด popup สำหรับกรอกเลขบัตรประชาชน 13 หลัก โดยระบบจะตรวจสอบว่าเป็นตัวเลขเท่านั้น หากผู้ใช้พิมพ์ภาษาไทย ตัวอักษร หรืออักขระอื่นที่ไม่ใช่ตัวเลข ระบบจะแจ้งเตือนให้กรอกเฉพาะตัวเลข 0-9

### 2. หน้าวัดค่าสัญญาณชีพ

หน้านี้ใช้สำหรับควบคุมการวัดค่าจากอุปกรณ์จริง โดยมีปุ่มวัดค่าแยกตามประเภทอุปกรณ์ ได้แก่

- ปุ่มวัดความดันโลหิต
- ปุ่มวัดออกซิเจนในเลือด
- ปุ่มวัดอุณหภูมิ

เมื่อกดวัด ระบบจะแสดงสถานะบนปุ่ม เช่น กำลังวัด วัดแล้ว หรือวัดไม่สำเร็จ พร้อมแสดงค่าที่อ่านได้บนหน้าจอ ค่าที่แสดงจะแยกสีตามชนิดของข้อมูล เพื่อให้ผู้ใช้งานอ่านค่าได้ง่าย

### 3. หน้าสรุปผลการวัด

หน้านี้แสดงผลการวัดทั้งหมดในรูปแบบตาราง ได้แก่ ความดันโลหิต ชีพจร ออกซิเจนในเลือด และอุณหภูมิ มีปุ่มสำหรับบันทึกข้อมูลไปยัง Backend API และมีปุ่มดูข้อมูลย้อนหลัง 4 รายการล่าสุด

ข้อมูลย้อนหลังถูกจัดเป็นตาราง โดยมีคอลัมน์วันที่และเวลา ความดันโลหิต ชีพจร ออกซิเจนในเลือด และอุณหภูมิ หากข้อมูลบางรายการไม่มีค่า ระบบจะแสดงเป็น `-`

## Flow การใช้งาน

1. เปิดโปรแกรม
2. เสียบบัตรประชาชนและกดอ่านข้อมูล
3. หากอ่านบัตรไม่สำเร็จ สามารถกรอกเลขบัตรประชาชนเองผ่าน popup
4. ระบบเข้าสู่หน้าวัดค่าสัญญาณชีพ
5. ผู้ใช้กดวัดค่าจากอุปกรณ์แต่ละชนิด
6. ระบบแสดงผลการวัดบนหน้าจอ
7. ผู้ใช้กดดูสรุปผลการวัด
8. ผู้ใช้สามารถดูข้อมูลย้อนหลังหรือบันทึกข้อมูลไปยัง Backend
9. เมื่อบันทึกสำเร็จ ระบบแสดงข้อความสำเร็จและกลับสู่หน้าเริ่มต้น

## การแยกส่วนการทำงานของโค้ด

ระบบแยกหน้าที่หลักออกเป็น 2 ส่วน

### 1. ส่วน UI

ไฟล์ `carekeeper_ui.py` รับผิดชอบเรื่องหน้าจอทั้งหมด เช่น

- การสร้างหน้าอ่านบัตร หน้าวัดค่า และหน้าสรุปผล
- การเปิด popup กรอกเลขบัตรประชาชน
- การจัดการปุ่มวัดค่า
- การแสดงข้อความสำเร็จ/ผิดพลาด
- การแสดงข้อมูลย้อนหลัง
- การส่งคำสั่งไปยัง provider

### 2. ส่วน Provider

ไฟล์ `carekeeper_providers.py` รับผิดชอบการเชื่อมต่อข้อมูลและอุปกรณ์ เช่น

- อ่านข้อมูลบัตรประชาชน
- อ่านค่าความดันโลหิต
- อ่านค่า SpO2
- อ่านสถานะแบตเตอรี่
- สแกนและเชื่อมต่อ Wi-Fi
- สแกนและเชื่อมต่อ Bluetooth
- ส่งข้อมูลไปยัง Backend API
- เตรียมข้อมูลย้อนหลัง

การแยกส่วนนี้ช่วยให้สามารถพัฒนา UI ด้วยข้อมูลจำลองได้ โดยไม่ต้องต่ออุปกรณ์จริงตลอดเวลา และช่วยให้การเปลี่ยนอุปกรณ์ในอนาคตทำได้ง่ายขึ้น

## การจัดการ Thread / Process

Python มีข้อจำกัดเรื่อง Global Interpreter Lock (GIL) ทำให้โค้ด Python ที่ใช้ CPU หนัก ๆ ไม่ได้ประมวลผลหลาย thread พร้อมกันแบบเต็มประสิทธิภาพ แต่ระบบ Care Keeper ส่วนใหญ่เป็นงานแบบ I/O-bound เช่น รออ่านค่าจาก serial port, Bluetooth, Wi-Fi, smart card reader และ HTTP request ไปยัง Backend ดังนั้นการใช้ thread ยังเหมาะสมกับงานลักษณะนี้

ในโปรแกรมนี้ UI หลักของ PySide6 ทำงานอยู่บน main thread เพื่อให้หน้าจอวาดผลและรับการกดปุ่มได้ต่อเนื่อง ส่วนงานที่อาจใช้เวลานานจะถูกส่งไปทำใน `ProviderTask` ซึ่งสืบทอดจาก `QThread` เช่น

- อ่านข้อมูลบัตรประชาชน
- วัดความดันโลหิต
- วัดค่า SpO2
- วัดอุณหภูมิ
- ส่งข้อมูลไปยัง Backend
- สแกนและเชื่อมต่อ Wi-Fi
- สแกนและเชื่อมต่อ Bluetooth
- อ่านสถานะ Wi-Fi, Bluetooth และแบตเตอรี่เป็นรอบ ๆ

การแยกงานลักษณะนี้ช่วยลดโอกาสที่หน้าจอจะค้างระหว่างรออุปกรณ์จริงตอบกลับ และช่วยให้สามารถแสดงข้อความแจ้งเตือนผู้ใช้เมื่อเกิดข้อผิดพลาดได้

สำหรับ Wi-Fi และ Bluetooth มีการป้องกันเพิ่มเติม 2 ส่วน

1. มี `network_task` สำหรับกันไม่ให้ผู้ใช้เริ่มสแกนหรือเชื่อมต่อ Wi-Fi/Bluetooth ซ้อนกันหลายครั้ง
2. คำสั่งระบบ เช่น `nmcli`, `bluetoothctl`, `iwgetid` และ `ip link` ถูกกำหนด timeout เพื่อป้องกันกรณี subprocess ค้างบน Raspberry Pi

ถ้าในอนาคตมีงานที่ใช้ CPU หนัก เช่น ประมวลผลสัญญาณ ECG, วิเคราะห์กราฟแบบ real-time หรือประมวลผลข้อมูลจำนวนมาก ควรแยกออกเป็น process หรือ service แยกจาก GUI แต่สำหรับงานปัจจุบันที่เป็นการอ่านอุปกรณ์และรอผลลัพธ์ การใช้ `QThread` ร่วมกับ timeout ถือว่าเพียงพอและเหมาะกับการใช้งานบน Raspberry Pi

## ระบบ Retry และ Auto-disable อุปกรณ์ (`carekeeper_retry.py`)

อุปกรณ์จริง (บัตรประชาชน, ความดัน, SpO2) เชื่อมต่อผ่าน serial/BLE ซึ่งหลุดหรือไม่พร้อมได้บ่อย ระบบจึงห่อการเรียกอุปกรณ์แต่ละครั้งด้วย `retry_with_notify` (และ `retry_with_notify_async` สำหรับ BLE)

- พยายามเรียกซ้ำสูงสุด 3 ครั้ง (ปรับได้) แบบ linear backoff ระหว่าง attempt
- มี `SubsystemRegistry` เก็บสถานะของแต่ละอุปกรณ์ (`wifi`, `bluetooth`, `idcard`, `bp_monitor`, `spo2`) เป็น process-wide singleton
- ถ้าครบจำนวนครั้งแล้วยังไม่สำเร็จ subsystem นั้นจะถูก `disable()` และ UI จะอัปเดตสถานะ/ปุ่มให้ผู้ใช้เห็นว่าอุปกรณ์นี้ใช้งานไม่ได้ชั่วคราว
- รอบถัดไปที่ผู้ใช้กดใช้งานอุปกรณ์นั้นอีกครั้ง ระบบจะ `enable()` กลับมาลองใหม่โดยอัตโนมัติ

## คิวส่งข้อมูลออฟไลน์ (`carekeeper_queue.py`)

เพื่อรองรับกรณีอินเทอร์เน็ต/Wi-Fi หลุดตอนกดบันทึกผลตรวจ ระบบมี `SubmissionQueue` ซึ่งเป็นคิวแบบ FIFO เก็บลง SQLite ที่ `data/carekeeper_queue.db`

- เมื่อส่งข้อมูลไป Backend ไม่สำเร็จ payload จะถูก `enqueue()` ลงไฟล์แทนการทิ้งข้อมูล จึงรอดจากการปิด/รีสตาร์ทโปรแกรม
- `QueueDrainWorker` เป็น background thread ที่ poll คิวทุก 5 วินาที ถ้า Wi-Fi กลับมาออนไลน์ (เช็คผ่าน `is_online_fn`) จะส่งรายการที่ค้างอยู่ตามลำดับเดิม ส่งสำเร็จลบทิ้งทันที ส่งไม่สำเร็จเพิ่ม `attempts` แล้วลองใหม่รอบถัดไป
- ถ้าโปรแกรมถูกปิดกลางคันตอนกำลังส่ง (`status='sending'`) จะถูก reset เป็น `pending` ใหม่ตอนเริ่มโปรแกรมรอบหน้า กันไม่ให้ข้อมูลตกหายแบบเงียบ ๆ

## โหมดการทำงาน

### Mock UI Mode

ใช้สำหรับทดสอบหน้าจอโดยไม่ต้องต่ออุปกรณ์จริง เหมาะสำหรับออกแบบ UI และทดสอบ flow การใช้งาน

```bash
python main_demo.py
```

โหมดนี้ใช้ `MockCareKeeperProvider` เพื่อสร้างข้อมูลตัวอย่าง เช่น ข้อมูลผู้รับบริการ ค่าสัญญาณชีพ สถานะแบตเตอรี่ และข้อมูลย้อนหลัง

### Real Hardware Mode

ใช้สำหรับรันบน Raspberry Pi ที่เชื่อมต่ออุปกรณ์จริง

```bash
python main_real.py
```

โหมดนี้ใช้ `RealCareKeeperProvider` หากอุปกรณ์ใดไม่พร้อมใช้งาน ระบบจะแสดงข้อความแจ้งเตือนบนหน้าจอ

| วัดชีพจร | อ่านจากเครื่องวัดความดัน | `lib/bp_monitor.py` |

| วัดอุณหภูมิ | ยังเป็น placeholder ใน real provider | ต้องเพิ่มโมดูลวัดอุณหภูมิจริง |
| อ่านแบตเตอรี่ | มี path อ่านจาก UPS แล้ว | `lib/ups.py` |
| Wi-Fi | ใช้ `nmcli` สำหรับสแกนและเชื่อมต่อ | Raspberry Pi/Linux |
| Bluetooth | ใช้ `bluetoothctl` สำหรับสแกนและเชื่อมต่อ | Raspberry Pi/Linux |
| ส่งข้อมูล Backend | ส่งผ่าน HTTP POST แล้ว | `requests.post()` |
| ข้อมูลย้อนหลัง | Mock พร้อมแล้ว, real ยังรอ GET API | `get_measurement_history()` |

ข้อสำคัญ: ในสถานะปัจจุบันส่วนอุณหภูมิของโหมดอุปกรณ์จริงยังไม่สมบูรณ์ เพราะ `RealCareKeeperProvider.measure_temperature()` ยังเป็น `NotImplementedError` จึงต้องเพิ่มโมดูลอ่านค่าอุณหภูมิจริงก่อนนำไปใช้งานเต็มรูปแบบ

## การปรับแต่งระบบปฏิบัติการบน Raspberry Pi 5 (Kiosk Setup)

นอกจากโค้ด Python แล้ว เครื่องตรวจจริงต้องปรับแต่งฝั่ง OS (Raspberry Pi OS) เพื่อให้บูตขึ้นมาแล้วรันโปรแกรม CareKeeper แบบ kiosk ได้เอง โดยไม่ต้องมีคนนั่ง login/เปิดโปรแกรมเอง งานส่วนนี้ทำนอก repo (แก้ที่ตัวเครื่อง Pi) ยกเว้นไฟล์ตัวอย่าง systemd ที่เก็บไว้ใน `conf/`

| งาน | รายละเอียด | อ้างอิงไฟล์ |
| --- | --- | --- |
| Auto-run โปรแกรมตอนบูต | systemd service สั่งรัน `main_real.py` ด้วย Python venv ของ CareKeeper ทันทีที่เข้า graphical target และรอ `pcscd` พร้อมก่อน, ตั้ง `Restart=on-failure` ให้ฟื้นเองถ้าโปรแกรม crash | `conf/carekeeper.service` |
| Restart เครื่องทุกคืน | timer สั่ง restart service ทุกวันตี 3 เพื่อล้าง memory/cache ไม่ให้สะสมจนเครื่องช้าเมื่อรันต่อเนื่องยาว ๆ | `conf/carekeeper-restart.service`, `conf/carekeeper-restart.timer` |
| จำกัดขนาด log | ตั้ง `systemd-journald` ให้ใช้ log ได้ไม่เกิน 200MB และเก็บย้อนหลังไม่เกิน 14 วัน กัน disk เต็มจาก log สะสม | `conf/carekeeper-journal.conf` |
| สิทธิ์เข้าถึงเครื่องอ่านบัตร/USB sensor | เพิ่ม udev rule ให้ USB ของเครื่องอ่านบัตร/อุปกรณ์ sensor เข้าถึงได้โดยไม่ต้องเป็น root (`MODE="0666"`) | ทำที่ `/etc/udev/rules.d/99-thaiidcard.rules` บนตัวเครื่อง |
| อ่านบัตรได้แม้ไม่มีคน login หน้าจอ | เพิ่ม polkit rule ให้ `pcscd` (PC/SC daemon ของเครื่องอ่านบัตร) อนุญาต client ที่ไม่มี active session ได้ เพราะ kiosk service รันแบบไม่ผูกกับ session ผู้ใช้ที่ login จริง | ทำที่ `/etc/polkit-1/rules.d/50-pcscd.rules` บนตัวเครื่อง |
| ซ่อน desktop/taskbar ตอน kiosk | แก้ autostart ของ labwc (Wayland compositor บน Raspberry Pi OS รุ่นใหม่) ให้ไม่เปิด file manager/taskbar เหลือไว้แค่สิ่งที่จำเป็น | ทำที่ `/etc/xdg/labwc/autostart` และ `~/.config/labwc/autostart` บนตัวเครื่อง |
| ซ่อนข้อความ "Welcome to the Raspberry Pi Desktop" ตอนบูต | แทนที่ภาพ splash ของ Plymouth ด้วยพื้นสีเดียวกับพื้นหลัง เพื่อไม่ให้เห็นข้อความ/โลโก้ตอนเปิดเครื่องก่อนเข้าโปรแกรม | แก้ที่ `/usr/share/plymouth/themes/pix/splash.png` บนตัวเครื่อง (มี backup `.png.bak`) |

ลำดับการติดตั้งแบบเต็ม (ตั้งแต่ raspi-config, ติดตั้ง package ระดับ OS, สิทธิ์ user, ไปจนถึง reboot ตรวจสอบ checklist) อยู่ในเอกสารแยก **CareKeeper Pi5 Setup Runbook** ที่ทีมใช้อ้างอิงตอน setup เครื่องใหม่ ควรเก็บไฟล์นี้คู่กับ repo (เช่นใน `docs/`) เพื่อให้ใครตั้งเครื่องใหม่ตามได้โดยไม่ต้องถามทีมเดิม

## รูปแบบข้อมูลที่ส่งไป Backend

เมื่อผู้ใช้กดบันทึกข้อมูลในหน้าสรุป ระบบจะส่งข้อมูลแบบ JSON ไปยัง Backend API

```json
{
  "mac": "11.11.11.11",
  "spo2": 98,
  "heart_rate": 70,
  "pr_bpm": 70,
  "sys": 120,
  "dia": 78,
  "pulse": 70
}
```

หมายเหตุ:

- `mac` จะพยายามอ่านจาก MAC address ของ Raspberry Pi ก่อน
- `heart_rate`, `pr_bpm` และ `pulse` ใช้ค่าชีพจรชุดเดียวกันตามรูปแบบ API ปัจจุบัน
- API key ปัจจุบันเป็นค่าทดสอบ `test`
- หากใช้งานจริงควรย้ายค่า API URL และ API key ไปไว้ใน `.env` หรือ configuration ที่ปลอดภัยกว่า

## การจัดการข้อผิดพลาด

ระบบมีการแจ้งเตือนผู้ใช้ในกรณีสำคัญดังนี้

| กรณี | วิธีแจ้งเตือน |
| --- | --- |
| อ่านบัตรไม่สำเร็จ | เปิด popup กรอกเลขบัตรเอง และแสดงข้อความผิดพลาด |
| กรอกเลขบัตรไม่ครบ | แสดงข้อความใน popup และ toast แจ้งเตือน |
| กรอกตัวอักษร/ภาษาไทยในเลขบัตร | แสดงข้อความให้กรอกเฉพาะตัวเลข 0-9 |
| วัดความดันไม่สำเร็จ | ปุ่มเปลี่ยนเป็นสถานะวัดไม่สำเร็จและแสดง toast |
| วัด SpO2 ไม่สำเร็จ | ปุ่มเปลี่ยนเป็นสถานะวัดไม่สำเร็จและแสดง toast |
| วัดอุณหภูมิไม่สำเร็จ | ปุ่มเปลี่ยนเป็นสถานะวัดไม่สำเร็จและแสดง toast |
| ส่งข้อมูล Backend ไม่สำเร็จ | แสดงข้อความผิดพลาดและให้กดบันทึกใหม่ |

ข้อสังเกต: การตรวจสถานะ Wi-Fi/Bluetooth/แบตเตอรี่ที่ทำงานเป็นรอบ ๆ จะไม่ popup รบกวนผู้ใช้ทุกครั้งหากอ่านสถานะไม่ได้ เพื่อไม่ให้หน้าจอเด้งเตือนซ้ำระหว่างใช้งาน

## สิ่งที่ปรับปรุงล่าสุด

- เพิ่มระบบ retry + auto-disable subsystem (`carekeeper_retry.py`) สำหรับบัตรประชาชน, ความดัน, SpO2, Wi-Fi, Bluetooth เวลาเชื่อมต่ออุปกรณ์ไม่สำเร็จ
- เพิ่มคิวส่งข้อมูลออฟไลน์แบบ SQLite (`carekeeper_queue.py`) พร้อม background worker คอย drain คิวอัตโนมัติเมื่อกลับมาออนไลน์
- เพิ่ม logging กลางของระบบ (`carekeeper_logging.py`) และ log thread identity เพื่อ debug ปัญหา thread/Qt
- เพิ่มชุด pytest ครอบคลุม provider, queue, retry, threading และ UI flow หลัก (โฟลเดอร์ `tests/`)
- เพิ่ม systemd unit/timer สำหรับรันแบบ kiosk, restart เครื่องตอนตี 3 และจำกัดขนาด log บน Raspberry Pi (โฟลเดอร์ `conf/`)
- เพิ่ม `QThread` สำหรับงาน Wi-Fi/Bluetooth แยกจาก UI และกันไม่ให้สแกน/เชื่อมต่อซ้อนกัน
- เพิ่ม timeout ให้คำสั่งระบบ `nmcli`, `bluetoothctl`, `iwgetid` และ `ip link` เพื่อลดปัญหาหน้าจอค้างบน Raspberry Pi
- เปลี่ยนการกรอกเลขบัตรประชาชนเองให้เป็น popup/dialog
- เพิ่มการแจ้งเตือนเมื่อผู้ใช้กรอกตัวอักษรหรือภาษาอื่นในช่องเลขบัตร
- ปรับช่องกรอกรหัสผ่านของ Wi-Fi ให้ตัวอักษรเป็นสีดำ มองเห็นได้บนพื้นหลังสีขาว
- ปรับข้อมูลย้อนหลังเป็นตาราง 4 รายการล่าสุด ไม่ใช้ scroll เพื่อให้เหมาะกับหน้าจอ Raspberry Pi
- แยก style หลักไว้ใน `carekeeper_style.py`
- แยก provider สำหรับ mock และ real hardware ใน `carekeeper_providers.py`

## ข้อจำกัดปัจจุบันและงานที่ควรทำต่อ

1. **อุณหภูมิในโหมดจริงยังไม่เชื่อมต่อ**
   ต้องเพิ่มโมดูลอ่านค่า temperature sensor จริง แล้วแก้ `RealCareKeeperProvider.measure_temperature()`

2. **ข้อมูลย้อนหลังจริงยังไม่มี GET API**
   ตอนนี้ mock แสดงได้แล้ว แต่โหมดจริงยังรอ endpoint จาก Backend

3. **API key ยัง hardcode**
   เพื่อความสะดวกในการทดสอบ ปัจจุบัน API key ยังอยู่ในโค้ด ควรย้ายไป `.env` ก่อนใช้งานจริง

4. **ควรทดสอบบน Raspberry Pi หลังแก้ UI ทุกครั้ง**
   เพราะหน้าจอมีขนาดเฉพาะ และ touchscreen/on-screen keyboard อาจแสดงผลต่างจาก Windows

5. **log ปัจจุบันยังพิมพ์ไปที่ console เท่านั้น**
   `carekeeper_logging.py` ตั้ง logging ให้แล้ว แต่ยังไม่ได้เขียนเป็นไฟล์/หมุน log บนตัวเครื่อง ควรต่อเข้ากับ `journald` (มี `conf/carekeeper-journal.conf` จำกัดขนาดไว้แล้ว) หรือเขียนไฟล์ log แยกเพื่อตรวจสอบย้อนหลังได้ง่ายขึ้น

6. **เอกสาร OS setup ยังไม่อยู่ใน repo**
   ขั้นตอนปรับแต่ง Raspberry Pi 5 (raspi-config, udev, polkit, labwc autostart, Plymouth splash) ปัจจุบันอยู่ในเอกสารแยกนอก repo ควร commit runbook นี้เข้ามาเก็บไว้ใน `docs/` เพื่อให้ตั้งเครื่องใหม่ได้โดยไม่ต้องถามทีมเดิม

## สรุปสำหรับการนำเสนอ

Care Keeper เป็น GUI สำหรับเครื่องตรวจวัดสัญญาณชีพบน Raspberry Pi ที่ออกแบบให้ผู้ใช้ทั่วไปใช้งานได้ง่าย ระบบเริ่มจากการอ่านข้อมูลบัตรประชาชนหรือกรอกเลขบัตรเอง จากนั้นเข้าสู่หน้าวัดค่าสัญญาณชีพ แสดงค่าความดันโลหิต ชีพจร SpO2 และอุณหภูมิ ก่อนสรุปผลและส่งข้อมูลไปยัง Backend

โครงสร้างโปรแกรมแยกส่วน UI ออกจาก provider ทำให้สามารถทดสอบด้วย mock data ได้ และสามารถเชื่อมต่ออุปกรณ์จริงบน Raspberry Pi ได้โดยใช้ provider อีกชุดหนึ่ง ปัจจุบันส่วนอ่านบัตร ความดัน SpO2 แบตเตอรี่ Wi-Fi Bluetooth และการส่ง Backend มีทางเชื่อมต่อแล้ว และมีชั้นความทนทานเพิ่มเข้ามา ได้แก่ ระบบ retry + auto-disable อุปกรณ์ที่เชื่อมต่อไม่สำเร็จ และคิวส่งข้อมูลแบบออฟไลน์ที่ค้างไว้ในเครื่องจนกว่าจะส่งสำเร็จ พร้อมชุด pytest ครอบคลุม provider/queue/retry/UI หลัก

นอกจากโค้ดแอป ยังมีงานฝั่ง OS บน Raspberry Pi 5 เพื่อให้เครื่องบูตเข้าโปรแกรมแบบ kiosk ได้เอง เช่น systemd service/timer สำหรับ auto-run และ restart เครื่องตอนตี 3, จำกัดขนาด log, ปรับสิทธิ์ udev ให้เครื่องอ่านบัตร, แก้ polkit ให้ `pcscd` ใช้งานได้แม้ไม่มี session login, ซ่อน taskbar/file manager ผ่าน labwc autostart และซ่อนข้อความตอนบูตของ Plymouth — รายละเอียดอยู่ในหัวข้อ "การปรับแต่งระบบปฏิบัติการบน Raspberry Pi 5" ด้านบน

ส่วนที่ยังต้องพัฒนาต่อคือโมดูลวัดอุณหภูมิจริง, API สำหรับดึงข้อมูลย้อนหลังจาก cloud, การเขียน log ลงไฟล์/journald อย่างเป็นระบบ และการนำเอกสาร OS setup runbook เข้ามาเก็บไว้ใน repo
